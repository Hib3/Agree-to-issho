# Original Algorithm Understanding

## 結論

原作アルゴリズムは、完全なバイナリ逆解析までは未完了。
ただし、現時点では次の構造として理解するのが最も妥当。

```text
PS本体側ローダー
  -> DOKODEMO.417 内部アーカイブを読む
  -> 中核PS-X EXE候補をロードする
  -> キャラ別/日別/会話別らしき内部リソース群を参照する
  -> Memory Card / PocketStation / 日付時計 / 日記 / 単語 / 質問状態を保存・復元する
```

## 確認済み

- ISO内の `SCPS_100.92` はPS-X EXEで、サイズは86,016 bytes。
- ISO内の `DOKODEMO.417` は先頭 `CNA` の内部アーカイブ候補。
- `DOKODEMO.417` には630件の内部エントリ表がある。
- `DOKODEMO.417` 内 entry `1` はPS-X EXE候補で、サイズは468,992 bytes。
- entry `1` には `SAVE`, `LOAD`, `CARD`, `MCRD`, `DIARY`, `WORD`, `QUEST`, `CLOCK`, `PDA` 系の英数字キーワードが集中している。
- entry `1` の `PDA`, `CLOCK`, `DIARY` 系キーワードの一部は、MIPS命令側から仮想アドレス参照されている。
- entry `1` 内の関数候補 `0x80019e2c..0x8001aa4c` は `PDA`, `DIARY`, `CLOCK`, `MEM` を同じ関数内で参照し、153回の関数呼び出しを含む大きめの処理塊。
- entry `1` 内の関数候補 `0x800345dc..0x800347dc` は `DIARY` を参照する小さめの処理塊。
- entry `1` 内の関数候補 `0x80045d5c..0x80045dbc` と `0x800481fc..0x800482e8` は `LOAD` を参照する小さめの処理塊。
- entry `1` 呼び出しグラフ上でも、`PDA/CLOCK/MEM/DIARY` 関数候補は直接呼び出し153回・ユニーク呼び先30件で、同期/初期化系の大きな制御関数らしい。
- `LOAD` 関数候補2件は直接呼び出し1回/4回で、ロード処理内の小さな復元ステップらしい。
- `WORD`, `QUEST`, `ANS` はentry `1`内にASCII構造ラベルとして存在するが、今回の直接参照/ポインタ表解析ではコード参照を確認できていない。
- 範囲クラスタ解析では、`WORD` 5件、`QUEST` 22件を含むentry `1`内の領域 `0x80010f4c..0x8001174b` がコードから90回参照される。
- これにより、単語/質問系ラベルを含む参照済み領域の存在は確認済み。ただし学習アルゴリズム本体かどうかは未確認。
- `ANS` 2件を含むentry `1`内の領域 `0x80010a50..0x80010ea7` はコードから34回参照される。
- `WORD/QUEST`領域と`ANS`領域は主な参照元関数群が異なるため、質問系と回答/周辺処理系が分かれている可能性がある。
- 両関数群は共有関数0件だが、直接リンク5件と共通呼び先9件があるため、別責務だが連携するサブシステムと見るのが自然。
- 自然文らしい候補はCNA entry `1` が最多で、entry `1` は中核処理だけでなく短い自然文/構造文候補も含む。
- `%s/%d/%c` などのフォーマット記号候補はentry `1` に集中するが、ユーザー学習語の差し込み用途かは未確認。
- entry `1` 内の`%s`は42件あり、18件がMIPSコードから参照される。文字列スロット整形は確認済み。
- entry `43, 46, 49, 52, 55` はサイズが約320KBで、短いShift_JIS候補が約5,000件ずつあり、5本で形が揃っている。
- ただし entry `43, 46, 49, 52, 55` は先頭64 bytesが一致し、ゼロ間隔に16 bytes周期が強いため、辞書ではなく画像/表形式バイナリによる文字列誤検出の可能性が高い。
- index `1000`, `2000`, `3000`, `4000`, `5000` 台には、似たサイズ・似た文字列密度のリソース群が並ぶ。

## 不明

- `CNA` の正式フォーマット名。
- `LPF` 候補の正式フォーマット名。
- entry `1` がPocketStation転送用か、PS本体側のサブプログラムか、または両方を含むか。
- entry `43, 46, 49, 52, 55` が画像、表、辞書、会話のどれか。現時点では画像または表形式バイナリ寄り。
- セーブデータ内の実際のフィールド配置。
- 単語カテゴリ、好感度、日記、質問状態の具体的なバイナリ構造。
- `WORD`, `QUEST`, `ANS` の実処理関数、テーブル、状態遷移の場所。
- `WORD/QUEST`参照済み領域の役割。表示ラベル、状態名、デバッグ名、質問テーブルのいずれか。
- `ANS`参照済み領域の役割。回答処理、カード/メモリ関連表示、状態名のいずれか。
- 自然文候補と学習会話/質問/発言テンプレートの対応。
- `%s` 等のフォーマット記号と学習語差し込みの対応。

## 推測: 原作の大まかな状態機械

この節は、公開されているゲーム機能とentry `1`の構造ラベルから組んだ作業仮説。
`WORD/QUEST`を含む参照済み領域は確認できたが、内部処理位置は未確認なので、原作実装そのものとしては扱わない。

```text
起動
  -> メモリーカード / PocketStation 状態確認
  -> 保存データロード
  -> 時計差分を反映
  -> キャラ状態復元
  -> 今日の会話/質問/日記候補を選ぶ

通常会話
  -> 現在状態から会話種別を選ぶ
  -> 覚えた単語候補を選ぶ
  -> キャラ/日付/イベント/親密度に応じて文脈を選ぶ
  -> 発話
  -> プレイヤー回答を受ける
  -> 感情・親密度・質問状態を更新

単語学習
  -> プレイヤーが言葉を入力
  -> 既存語か確認
  -> その言葉の種類や好き嫌いを質問
  -> 単語レコードに属性を付ける
  -> 会話・日記・ミニゲームで使える候補に入れる

保存
  -> キャラ状態
  -> 覚えた単語
  -> 単語属性
  -> 今日の進行
  -> 日記
  -> PocketStation側の持ち出し状態
  を保存する
```

## 推測: 単語レコードの構成

原作の実フィールドは不明。
ただし機能上、少なくとも次のような情報が必要になる。

- 表示語
- 読みまたは並び順判定用情報
- カテゴリ
- 好き嫌い/重要度
- 誰から教わったか、または入力由来
- 使用済み回数
- 最近使ったか
- 会話/日記/しりとりで使えるか

## 推測: 単語使用アルゴリズム

原作の正確な選択式、重み、乱数、クールダウンは不明。
下記は「覚えた言葉を会話に使う」という公開機能を満たすための抽象であり、原作ロジックの復元ではない。
コード参照される`%s`があるため、何らかのスロット式表示/整形は使われている。
ただし、ユーザー語差し込み専用とは断定しない。

```text
候補語 = 覚えた語から選ぶ
候補語をカテゴリで絞る
最近使った語を避ける
好感度/日付/会話種別に合う語を上げる
完全固定ではなくランダム性を混ぜる
選ばれた語を会話または日記の抽象スロットに差し込む
使用履歴を更新する
```

## 推測: セーブ/ロード方式

原作はPocketStationをメモリーカード兼携帯端末として使うため、通常のセーブだけでなく、持ち出し中の状態同期が必要。

```text
PS本体側
  - 大きい会話/表示/部屋/日記処理
  - ディスク上リソース参照
  - PocketStationとの同期

PocketStation側
  - 短い会話
  - 単語入力
  - 選択肢回答
  - 時計連動
  - 最小限のキャラ状態保存

同期時
  - 新しく教えた語を統合
  - 質問回答を統合
  - 日付/時刻差分を反映
  - キャラ状態と日記候補を更新
```

## 推測: 実装上の中心

`SCPS_100.92` はローダー寄りで、`DOKODEMO.417` entry `1` が実質的な中核処理を持つ可能性が高い。
理由は、entry `1` がPS-X EXE候補であり、保存、ロード、カード、日記、単語、質問、時計、PDA系キーワードをまとめて含み、その一部がコードから参照されているため。

このため、原作の学習処理は次のように分かれていると考える。

```text
中核処理 entry 1
  - セーブ/ロード境界
  - PocketStation/PDA境界
  - 時計取得
  - 日記生成/保存
  - 質問状態候補
  - 覚えた言葉の管理候補

CNA内リソース群
  - キャラ別または日別の会話素材
  - 表示/画像/音声制御素材
  - 質問/日記/イベント用の補助データ
```

特に `PDA + CLOCK + MEM + DIARY` が同じ大きな関数候補に集まるため、原作は「携帯端末状態を読む」「時計差分を見る」「メモリ状態を確認する」「日記や日次状態へ反映する」を一連の同期処理として扱っている可能性が高い。

## 推測: 処理分割

```text
大きな同期/初期化処理
  evidence: PDA/CLOCK/MEM/DIARY参照、直接呼び出し153回
  role:
    - PocketStation/PDA状態確認
    - 時計/日付確認
    - メモリ状態確認
    - 日記または日次状態へ反映
    - 複数サブ処理の制御

日記処理
  evidence: DIARY参照、直接呼び出し20回
  role:
    - 日記関連の表示/生成/保存の一部

ロード復元処理
  evidence: LOAD参照、直接呼び出し1回/4回
  role:
    - 保存データまたはリソースロード後の小さな復元ステップ
```

この分割はPWAでは、`bootSyncPipeline()`, `applyDailyDiaryState()`, `restoreLoadedState()` のような責務分離に対応させる。

## PWAで採用する理解

原作を再現するのではなく、確認できた機能境界から外れない抽象機構だけを採用する。
未確認の内部重み・乱数・会話テンプレート構造は採用しない。

```text
learnedWords
  -> category
  -> confidence
  -> likeScore
  -> usageCount
  -> lastUsedAt
  -> cooldownUntil

conversationState
  -> pendingQuestion
  -> recentTopics
  -> recentWordIds
  -> currentDay

portableState
  -> offline learned words
  -> offline answers
  -> lastSyncedAt

load()
  -> schema migration
  -> clock/day diff
  -> portable merge
  -> rebuild conversation queue

save()
  -> transactional IndexedDB write
  -> snapshot backup
```
